# 错误记录 — 管理类知识库 RAG 系统

> 本文档记录开发过程中出现过的错误、踩坑与修正，供后续参考。

---

## 错误 1：重复向量化导致效率极低

**现象**：每次运行评估脚本，系统都重新执行向量化（约40-50分钟），即使缓存已存在。

**根因**：
- `config.py` 中 `VECTOR_STORE_DIR` 指向的缓存目录与实际缓存位置不一致
- 评估脚本通过动态替换 `config` 模块的方式切换配置，导致 `main.py` 中的缓存检查路径与实际存储路径不匹配
- `_check_cache()` 方法检查的路径与 `save()` 方法保存的路径不同

**修正**：
1. 统一缓存目录为 `vector_store/`，不再区分版本
2. 评估脚本直接使用 `src/config.py`，不再动态替换模块
3. `build_knowledge_base()` 默认 `force_rebuild=False`，优先复用缓存
4. 只有显式传入 `--force-rebuild` 时才重新向量化

**规范**：**每次运行必须优先复用缓存，避免重复向量化。如需重建，必须显式指定 `--force-rebuild`。**

## 错误 2：未使用 FAISS 作为向量数据库

**现象**：早期版本使用 TF-IDF 向量化 + 简单余弦相似度检索，未使用 FAISS。

**根因**：
- 对作业要求理解不深，未注意到"推荐使用 FAISS 或者 ChromaDB"的要求
- TF-IDF 向量化召回率仅 61.7%（Top-3），远低于语义向量化

**修正**：
1. 将 TF-IDF 替换为 BGE-large-zh-v1.5 语义向量模型
2. 使用 FAISS `IndexFlatIP` 作为向量索引（内积检索，等价于余弦相似度）
3. 实现双粒度 FAISS 索引（small + large）
4. 向量存储支持缓存到本地文件（`faiss.index` + `metadata.json`）

**当前状态**：FAISS 已作为核心向量数据库，所有向量存储和检索均通过 FAISS 完成。

## 错误 3：有 Python 3.12 却以为没有

**现象**：项目目录下已有 `python312/` 内嵌环境，但仍然尝试使用系统 Python 3.15（alpha版），导致 faiss-cpu 安装失败。

**根因**：
- 未检查项目目录下已有的 Python 环境
- 盲目使用系统默认 Python 版本

**修正**：
1. 明确使用项目内嵌的 `python312/python.exe` 运行所有脚本
2. faiss-cpu 仅支持 Python 3.8-3.12，不支持 3.13+

**规范**：**运行脚本时必须使用 `python312\python.exe`，不要使用系统 Python。**

## 错误 4：NDCG 计算值超过 1.0

**现象**：NDCG@10 计算结果为 1.57，超出理论范围 [0, 1]。

**根因**：
1. 使用 `2^rel - 1` 增益公式，当 rel > 1 时增益过大
2. 未对 `retrieved_docs` 去重，同一文档多次出现导致增益累加

**修正**：
1. 改用线性增益 `rel / log2(i + 2)`
2. 在评估脚本中对 `retrieved_docs` 使用 `dict.fromkeys()` 去重
3. 修正后 NDCG@10 = 0.79（合理范围）

## 错误 5：Context Utilization 始终为 0

**现象**：Layer 2 的 Context Utilization 指标始终为 0。

**根因**：评估数据集缺少 `relevant_chunk_ids` 字段，导致 chunk 级指标无法计算。

**修正**：运行自动标注脚本填充 `relevant_chunk_ids` 字段。

## 错误 6：Cross-Encoder 模型下载失败（已解决）

**现象**：启用 Cross-Encoder 精排时，`RuntimeError` 下载模型失败。

**根因**：网络环境不稳定，无法从 HuggingFace 下载 `bge-reranker-large` 模型。

**修正**：
1. 首次尝试时网络不稳定导致下载失败，临时设置 `USE_CROSS_ENCODER = False`
2. 后续网络恢复后重新下载成功，模型缓存至 `models/BAAI/bge-reranker-large/`
3. 配置已改为 `USE_CROSS_ENCODER = True`，Cross-Encoder 精排已正式启用

**当前状态**：✅ 已解决。Cross-Encoder 精排正常运行，对检索结果进行二次精排，显著提升排序质量。

## 错误 7：PyTorch DLL 加载失败

**现象**：安装 torch 后运行时报错 `OSError: [WinError 126] 找不到指定的模块`。

**根因**：默认安装了 torch GPU 版，但 Windows 环境缺少 CUDA 运行时。

**修正**：
```bash
pip uninstall torch
pip install torch --index-url https://download.pytorch.org/whl/cpu
```

## 错误 8：向量存储保存路径不一致

**现象**：向量存储有时保存到项目目录，有时保存到系统临时目录。

**根因**：`vector_store.py` 的 `save()` 方法在主路径写入失败时，自动回退到 `tempfile.gettempdir()`，但 `load()` 方法优先检查备用路径，导致缓存位置不确定。

**修正**：
1. 确保主路径 `data/vector_store/` 目录有写入权限
2. `_check_cache()` 方法同时检查主路径和备用路径
3. 加载时优先使用主路径

---

## 错误 9：numpy 2.x 与 faiss 不兼容

**现象**：`No module named 'numpy._core'` 错误。

**根因**：numpy 2.x 重构了内部模块结构，faiss-cpu 的预编译 wheel 依赖 numpy 1.x 的 API。

**修正**：保持 numpy 1.x 版本，不要升级到 2.x。

---

## 错误 10：16份文档有2份无法加载

**现象**：16份文档中"管理思想史"和"高效管理的8020法则"无法加载。

**根因**：
1. "管理思想史"为扫描版DOCX，包含大量图片但无可提取文本
2. "高效管理的8020法则"因内容去重逻辑误判被删除

**修正**：
1. 删除扫描版DOCX文件（无法提取文本，对检索无贡献）
2. 检查去重逻辑，确保正常文档不被误删
3. 增强DOCX解析：三重fallback（python-docx → docx2txt → zip-xml提取），并检测扫描版DOCX

**当前状态**：✅ 已解决。15份文档全部正常加载。

## 错误 11：代码重复与日志不统一

**现象**：`retriever.py` 和 `metrics.py` 中各自定义了 `STOP_WORDS`，日志使用 `print` 散落各处。

**根因**：早期开发时各模块独立编写，未统一提取公共常量和日志。

**修正**：
1. 新增 `src/utils/constants.py`：统一存储 `STOP_WORDS` 和 `QUESTION_TYPE_PATTERNS`
2. 新增 `src/utils/logger.py`：提供 `get_logger()` 统一日志函数
3. `retriever.py` 和 `metrics.py` 改为从 `constants.py` 导入常量
4. 后续逐步将 `print` 替换为 `logger.info/warning/error`

**当前状态**：✅ 已解决。常量统一管理，日志系统已建立。

## 优化记录

### 优化 1：Cross-Encoder 精排

**目标**：提升检索结果排序质量，使相关片段排名更靠前。

**方案**：在 RRF 融合 + MMR 重排之后，增加 Cross-Encoder 精排阶段。

**实现**：
- 模型：`BAAI/bge-reranker-large`（560M参数，C-MTEB精排榜首）
- 位置：检索管道最后阶段，对 Top-K 候选进行二次打分
- 配置：`USE_CROSS_ENCODER = True`，`CROSS_ENCODER_TOP_K = 15`，`CROSS_ENCODER_BATCH_SIZE = 8`

**效果**：Cross-Encoder 将查询和文档拼接输入，计算精确相关性分数，比单纯向量相似度更准确。

### 优化 2：查询扩展增强

**目标**：提高查询召回率，特别是对短查询和术语查询。

**方案**：根据问题类型自动扩展查询词。

**实现**：
- 识别问题类型（定义/对比/方法/原因/案例）
- 根据类型追加领域关键词（如定义类→"概念 定义 含义"，对比类→"异同 对比分析 区别"）
- 生成多种查询变体（去停用词版、关键词组合版、领域扩展版）

**效果**：查询"什么是SWOT分析"自动扩展为6种变体，覆盖更多检索路径。

### 优化 3：BM25 章节标题 Boost

**目标**：对章节标题匹配查询的文档给予额外权重。

**方案**：在 BM25 检索中，对章节标题与查询词重叠的 chunk 给予分数加成。

**实现**：
- 提取每个 chunk 的章节标题
- 计算查询词与章节标题的词重叠数
- 重叠词越多，boost 越高（1 + overlap_count * 0.15）

**效果**：章节标题匹配的文档排名提升，更容易被检索到。

### 优化 4：MMR 动态参数

**目标**：根据问题类型动态调整 MMR 的相关性/多样性平衡。

**方案**：不同问题类型需要不同的多样性程度。

**实现**：
- 对比类问题：λ=0.30（更高多样性，需要多角度信息）
- 定义类问题：λ=0.60（更高相关性，需要精准定义）
- 默认：λ=0.45（平衡模式）

**效果**：对比类问题返回更多不同视角的片段，定义类问题返回更精准的定义。
