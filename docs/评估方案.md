# 管理类知识库 RAG 系统 — RAGAS风格评估方案

> 评估方法：RAGAS (Retrieval Augmented Generation Assessment) — Reference-Free
> 无需金标数据集，无需人工标注，无需外部API


## 一、评估体系总览

本系统采用 RAGAS 风格的 reference-free 评估方法，由 AI 语义判断替代传统金标标注，直接评估 RAG 系统的检索质量。

### 核心思路

不需要预先标注的金标数据集。评估流程是：运行RAG系统检索 → AI判断每个检索结果的相关性 → 计算指标。AI语义判断能区分"论述"与"提及"、识别同义词、理解否定语境，远超关键词匹配。

### 评估指标

| 指标 | 含义 | 合格标准 |
|------|------|----------|
| **Context Precision@K** | 前K个chunk中相关chunk的比例 | @10 ≥ 0.65 |
| **Context Relevance** | 检索上下文与query的整体相关度 | ≥ 0.65 |
| **Context Sufficiency** | 上下文足以回答query的比例 | ≥ 0.75 |
| **Ranking Quality@K** | 相关chunk是否排在前面（MRR） | @10 ≥ 0.60 |
| **Context Utilization** | 相关chunk在检索结果中的占比 | ≥ 0.40 |
| **NDCG@K** | 考虑相关性分级的排序质量 | @10 ≥ 0.55 |

### 与RAGAS框架的对应关系

| RAGAS原版指标 | 本项目指标 | 适配说明 |
|--------------|-----------|----------|
| Context Precision | Context Precision@K | 基于AI判断的is_relevant |
| Context Relevancy | Context Relevance | 基于AI的context_relevance_score |
| Context Recall | Context Sufficiency | 用AI判断是否充分，无需reference answer |
| Faithfulness | 不适用 | 本系统不含LLM生成环节 |
| Answer Relevancy | 不适用 | 本系统不含LLM生成环节 |


## 二、评估流程

### 三步流程

```
Step 1: 收集检索结果
  python scripts/collect_retrieval_results.py
  → 输出: data/eval/retrieval_results.json

Step 2: AI评估（由IDE中的AI助手完成）
  AI读取检索结果，对每个query-chunk对进行语义评估
  → 输出: data/eval/ragas_evaluation.json

Step 3: 计算指标+生成报告
  python -m src.evaluation.eval_quick   # Quick模式
  python -m src.evaluation.eval_full    # Full模式
  → 输出: data/output/eval_ragas_*_report.json
```

### AI评估的数据格式

AI评估结果 `ragas_evaluation.json` 的结构：

```json
{
    "method": "ragas_style",
    "evaluator": "AI-Semantic-Judge-v1",
    "total_evaluated": 50,
    "evaluations": [
        {
            "query_id": "Q001",
            "query": "什么是FFS理论",
            "query_type": "definition",
            "difficulty": "easy",
            "context_relevance_score": 4,
            "is_sufficient": true,
            "chunk_evaluations": [
                {
                    "rank": 1,
                    "chunk_id": "xxx_chunk_1",
                    "is_relevant": true,
                    "relevance_score": 5,
                    "reason": "直接定义了FFS理论的核心概念"
                },
                {
                    "rank": 2,
                    "chunk_id": "xxx_chunk_2",
                    "is_relevant": false,
                    "relevance_score": 1,
                    "reason": "仅提及FFS但无实质论述"
                }
            ]
        }
    ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `context_relevance_score` | 1-5 | 检索上下文与query的整体相关度 |
| `is_sufficient` | bool | 检索到的上下文是否足以回答query |
| `chunk_evaluations[].is_relevant` | bool | 该chunk是否与query语义相关 |
| `chunk_evaluations[].relevance_score` | 1-5 | 该chunk的相关程度（5=直接回答，1=完全无关） |
| `chunk_evaluations[].reason` | str | 判断理由（可审计） |


## 三、指标体系详解

### 3.1 核心指标

| 指标 | 含义 | 计算方式 | 合格标准 |
|------|------|----------|----------|
| **Context Precision@K** | 前K个chunk中相关chunk的比例 | `relevant_count / K` | @10 ≥ 0.65 |
| **Context Relevance** | 检索上下文与query的整体相关度 | `avg(score) / 5.0` | ≥ 0.65 |
| **Context Sufficiency** | 上下文足以回答query的比例 | `sufficient_count / total` | ≥ 0.75 |
| **Ranking Quality@K** | 相关chunk是否排在前面（MRR） | `avg(1/first_relevant_rank)` | @10 ≥ 0.60 |
| **Context Utilization** | 相关chunk在检索结果中的占比 | `avg(relevant / total_chunks)` | ≥ 0.40 |
| **NDCG@K** | 考虑相关性分级的排序质量 | DCG/IDCG | @10 ≥ 0.55 |

### 3.2 总体评级

| 评级 | 条件 |
|------|------|
| A (优秀) | Precision@10 ≥ 0.80 且 Relevance ≥ 0.80 且 Sufficiency ≥ 0.90 |
| B+ (良好) | Precision@10 ≥ 0.65 且 Relevance ≥ 0.65 且 Sufficiency ≥ 0.75 |
| B (合格) | Precision@10 ≥ 0.50 且 Relevance ≥ 0.50 且 Sufficiency ≥ 0.60 |
| C (需改进) | 低于B标准 |

### 3.3 分组统计

按 `query_type`（definition/framework/application/comparison）和 `difficulty`（easy/medium/hard）分组计算核心指标，识别薄弱环节。

### 3.4 失败模式分析

| 编号 | 失败模式 | 诊断条件 |
|------|----------|----------|
| F1 | 零相关chunk | 所有检索chunk都不相关 |
| F2 | 低精确率 | 相关chunk占比 < 30% |
| F3 | 上下文不足 | is_sufficient = false |
| F4 | 排序靠后 | 第一个相关chunk排名 > 5 |
| F5 | 低相关度 | context_relevance_score < 3 |


## 四、评估Query集

50个query，覆盖知识库所有文档，4种类型：

| 类型 | 数量 | 示例 |
|------|------|------|
| definition | 15 | "什么是FFS理论"、"什么是OKR工作法" |
| framework | 20 | "OKR工作法的实施步骤是什么"、"如何进行有效的会议管理" |
| application | 8 | "如何成为卓有成效的管理者"、"如何应用8020法则提升管理效率" |
| comparison | 7 | "OKR和KPI有什么异同"、"X理论和Y理论的核心差异是什么" |

完整query列表见 `scripts/collect_retrieval_results.py` 中的 `EVAL_QUERIES`。

### Query 与知识库对齐原则

评估 query **必须**能在当前 15 份知识库文档中找到可检索的实质内容，否则低分反映的是 query 设计问题而非检索系统缺陷。设计 query 时应：

1. 先确认目标概念在 `data/knowledge_base/kb_contents.json` 或向量索引中有**正文论述**（非仅目录/标题提及）
2. 对比类 query 要求知识库中**同时存在**两个对比对象的论述段落
3. 若某 docx 经 30K 精炼后仅保留目录项（如原「三级火箭」query），应替换为同文档中有正文的 query（如「点线面体战略选择」）

> 示例：Q021 已从「三级火箭商业模式」调整为「点线面体战略选择的核心逻辑」，因前者在精炼后的知识库中仅有章节标题、无正文 chunk。

## 七、独立性保证

### 独立性审查结论

| 环节 | 是否独立 | 说明 |
|------|----------|------|
| Query设计 | 独立 | 50个query在脚本中硬编码，与QA库、TEST_QUESTIONS无重叠 |
| 检索结果收集 | 不适用 | 就是被评估系统自身的输出，Step 1需调用RAGPipeline是预期行为 |
| AI评估 | 独立 | AI语义判断不依赖任何检索组件（不用BGE、不用BM25、不用reranker、不用jieba） |
| 指标计算 | 独立 | metrics.py仅依赖numpy，纯数学计算，基于AI评估结果 |

### 模块依赖隔离

| 模块 | 依赖 | 是否依赖RAG组件 |
|------|------|----------------|
| `collect_retrieval_results.py` | RAGPipeline | 是（Step 1预期行为） |
| `dataset.py` | os, json, path_utils | 否 |
| `metrics.py` | numpy, typing | 否 |
| `reporter.py` | json, os | 否 |
| `eval_quick.py` | dataset, metrics, reporter | 否 |
| `eval_full.py` | dataset, metrics, reporter | 否 |

**关键设计**：评估指标计算模块（dataset/metrics/reporter/eval_quick/eval_full）与RAG系统完全解耦。唯一依赖RAGPipeline的 `collect_retrieval_results.py` 仅负责数据收集，其输出JSON是评估模块的唯一输入。
