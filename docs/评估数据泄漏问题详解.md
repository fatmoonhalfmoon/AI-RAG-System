# 评估数据泄漏问题：通俗讲解与修正方案

---

## 一、先搞懂几个关键术语

| 术语 | 通俗解释 |
|------|----------|
| **金标（Ground Truth）** | "标准答案"。评估时用来判断"检索结果对不对"的参照物。就像考试要有标准答案才能判分。 |
| **relevant_chunk_ids** | 当前项目中，每道评估题里标注的"哪些文本片段是正确答案"。 |
| **pipeline_generated** | 数据集里的 `gold_source` 字段值，意思是"这些标准答案是由系统流水线自动生成的"。 |
| **检索器（Retriever）** | 你的 RAG 系统中负责"从知识库里找出相关内容"的模块（BM25 + 向量检索 + RRF融合）。 |
| **Pooling（候选池合并）** | 用多套不同检索方法各自检索，把所有结果合并去重，形成一个大候选池，再交给人工审核。 |

---

## 二、你朋友说的"大问题"到底是什么？

### 2.1 用一个比喻来理解

想象你是一个老师，要考学生的数学水平。正常做法是：

> **正确流程**：先由专家独立出标准答案 → 学生做题 → 拿标准答案判分

但你现在做的事情等价于：

> **错误流程**：让学生自己做一遍题，把他的答案当成"标准答案" → 再让他做同一套题 → 拿他自己的答案判他自己的分

结果是什么？**学生当然接近满分**——因为"标准答案"就是他自己写的！

### 2.2 映射到你的项目

看你的 `eval_data/eval_dataset.json`，每道题都有：

```json
{
  "query_id": "Q001",
  "query": "管理的核心思想是什么",
  "relevant_chunk_ids": [
    "认识管理_small_chunk_16",
    "认识管理_small_chunk_82",
    "组织行为学_large_chunk_49",
    ...
  ],
  "gold_source": "pipeline_generated"   // ← 关键！
}
```

**`gold_source: "pipeline_generated"`** 的意思是：这些 `relevant_chunk_ids` 是你的检索系统自己跑出来的结果，不是人工标注的。

然后你的评估流程（`src/eval_full.py`）做的事情是：

1. 用同一个检索系统对同一个 query 再检索一遍
2. 拿检索结果和 `relevant_chunk_ids` 对比
3. 计算 recall、precision 等指标

**这就是循环论证**：用系统自己的输出当标准答案，再评估系统自己。指标当然会偏高。

### 2.3 具体看代码中的循环

在 `src/eval/metrics.py` 中：

```python
# Layer2Metrics.chunk_recall_at_k
@staticmethod
def chunk_recall_at_k(retrieved_chunks, relevant_chunk_ids, k=None):
    chunk_ids = [c.get("chunk_id", "") for c in retrieved_chunks]
    hits = sum(1 for cid in relevant_chunk_ids if cid in chunk_ids)
    return hits / len(relevant_chunk_ids)
```

这里 `relevant_chunk_ids` 来自数据集（pipeline 生成的），`retrieved_chunks` 来自同一套检索系统。如果两次检索结果高度重合（因为用的是同一套系统），recall 自然很高。

### 2.4 高估有多严重？

举个极端例子：

- 假设真正相关的 chunk 有 10 个
- 你的检索系统只能找到其中 5 个
- 但 `relevant_chunk_ids` 本身就只记录了这 5 个（因为金标就是系统自己生成的）
- 那么 recall = 5/5 = 100%，看起来完美！
- **但真实 recall = 5/10 = 50%**

差距就是这么大。

## 三、为什么这叫"信息泄漏"（Information Leakage）？

"泄漏"指的是：**被评估系统的信息，泄漏到了评估标准中**。

正常评估要求：**评估标准和被评估对象必须独立**。就像裁判不能同时是运动员。

在你的项目中：
- **被评估对象**：`HybridRetriever`（BM25 + 向量检索 + RRF 融合 + MMR + CrossEncoder 精排）
- **评估标准**：`eval_dataset.json` 中的 `relevant_chunk_ids`
- **问题**：`relevant_chunk_ids` 是由同一套检索流水线生成的

所以检索器的"偏好"（比如它倾向选哪些 chunk、用什么样的排序策略）已经嵌入到了金标中。评估时，系统当然会倾向于找回自己之前选过的 chunk。

## 四、正确的做法应该是什么？

### 4.1 黄金标准：人工标注

最理想的做法是：

1. 准备一批 query
2. 让人类标注者阅读所有知识库文档，标注每个 query 真正相关的 chunk
3. 用人工标注的结果作为金标

**优点**：完全独立，评估可信
**缺点**：成本高，你的知识库有 15 本书，人工全部阅读标注不现实

### 4.2 实际可行方案：Pooling + 人工审核

这是信息检索领域的标准做法（TREC 评测就是这么做的），步骤如下：

```
┌─────────────────────────────────────────────────────────────┐
│                    Pooling 标注流程                          │
│                                                             │
│  Query: "管理的核心思想是什么"                                │
│                                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  检索系统 A   │  │  检索系统 B   │  │  检索系统 C   │       │
│  │ (BM25 top10) │  │ (BGE top10)  │  │ (混合 top10)  │       │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘       │
│         │                  │                  │               │
│         ▼                  ▼                  ▼               │
│  ┌──────────────────────────────────────────────────┐       │
│  │           合并去重 = 候选池 (Pool)                  │       │
│  │  chunk_1, chunk_5, chunk_8, chunk_12, chunk_16,   │       │
│  │  chunk_23, chunk_45, chunk_67, ...               │       │
│  └──────────────────────┬───────────────────────────┘       │
│                         │                                   │
│                         ▼                                   │
│  ┌──────────────────────────────────────────────────┐       │
│  │           人工审核每个 chunk 是否相关               │       │
│  │  chunk_1: ✓ 相关                                   │       │
│  │  chunk_5: ✗ 不相关                                 │       │
│  │  chunk_8: ✓ 相关                                   │       │
│  │  ...                                              │       │
│  └──────────────────────┬───────────────────────────┘       │
│                         │                                   │
│                         ▼                                   │
│  ┌──────────────────────────────────────────────────┐       │
│  │  最终金标 = 人工确认相关的 chunk_ids               │       │
│  │  gold_source: "human"                             │       │
│  └──────────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

**关键点**：
- 用**多套不同**的检索方法（BM25、不同 embedding 模型、不同参数），确保候选池覆盖面广
- 合并去重后，每个候选 chunk 都由人工判断"相关/不相关"
- 最终金标的 `gold_source` 标记为 `"human"`

### 4.3 为什么 Pooling 能解决问题？

| 问题 | Pooling 如何解决 |
|------|-----------------|
| 单一系统视角遗漏 | 多套系统各有所长，合并后覆盖更全 |
| 信息泄漏 | 金标最终由人工决定，不依赖任何单一系统 |
| 评估不公平 | 所有被评估系统面对的是同一套人工金标 |

### 阶段一：立刻可做（代码防护，1-2小时）

#### 6.1 修改 `src/eval/dataset.py` — 加入金标来源校验

```python
def _validate(self):
    required_fields = ["query_id", "query", "query_type", "difficulty",
                       "relevant_docs", "answer_snippets", "reference_answer"]
    pipeline_count = 0
    human_count = 0
    for q in self.questions:
        for field in required_fields:
            if field not in q:
                raise ValueError(f"题目 {q.get('query_id', '?')} 缺少必填字段: {field}")
        source = q.get("gold_source", "unknown")
        if source == "pipeline_generated":
            pipeline_count += 1
        elif source == "human":
            human_count += 1

    total = len(self.questions)
    print(f"[数据集] 金标来源分布: 人工标注={human_count}, 流水线生成={pipeline_count}, 总计={total}")

    if pipeline_count > 0:
        print(f"[警告] {pipeline_count}/{total} 题的金标为 pipeline_generated，评估结果可能被高估！")
        print(f"[警告] 建议通过 Pooling + 人工审核重建金标后再运行评估。")
```

        for idx, score, meta in dense_results:
            candidate_ids.add(meta["chunk_id"])

        # 方法 C: 向量检索 top-20 (large chunks)
        # ... 同上

        # 方法 D: 原始 relevant_chunk_ids 也加入候选池
        for cid in q.get("relevant_chunk_ids", []):
            candidate_ids.add(cid)

        # 5. 导出候选池
        for cid in candidate_ids:
            chunk = find_chunk_by_id(all_chunks, cid)
            pool_records.append({

                "content_preview": chunk.get("content", "")[:200],
                "is_relevant": "",  # ← 人工填写
                "annotator": "",    # ← 人工填写
                "notes": "",        # ← 人工填写
            })

    # 6. 写出 CSV
    with open(output_csv, "w", newline="", encoding="utf-8-sig") as f:
        writer = csv.DictWriter(f, fieldnames=pool_records[0].keys())
        writer.writeheader()
        writer.writerows(pool_records)

    print(f"已导出 {len(pool_records)} 条候选到 {output_csv}")
    print(f"请人工填写 is_relevant 列（1=相关, 0=不相关）")

#### 6.5 人工标注流程

1. 打开导出的 CSV
2. 对每条记录，阅读 `content_preview`（或对照原文），判断该 chunk 是否与 query 相关
3. 在 `is_relevant` 列填写 `1`（相关）或 `0`（不相关）
4. 标注完成后，运行导入脚本将人工标注结果更新到 `eval_dataset.json`

#### 6.6 新增 `scripts/import_annotations.py`

```python
"""
读取人工标注完成的 CSV，更新 eval_dataset.json 中的金标。
"""
import json
import csv

def import_annotations(csv_path, dataset_path, output_path):
    # 1. 读取标注结果
    annotations = {}  # query_id -> [chunk_id, ...]
    with open(csv_path, "r", encoding="utf-8-sig") as f:
        reader = csv.DictReader(f)
        for row in reader:
            if row["is_relevant"] == "1":
                annotations.setdefault(row["query_id"], []).append(row["chunk_id"])

    # 2. 更新数据集
    with open(dataset_path, "r", encoding="utf-8") as f:
        data = json.load(f)

    updated_count = 0
    for q in data["questions"]:
        qid = q["query_id"]
        if qid in annotations:
            q["relevant_chunk_ids"] = annotations[qid]
            q["gold_source"] = "human"
            q.pop("needs_human_review", None)
            updated_count += 1

    # 3. 写出
    with open(output_path, "w", encoding="utf-8") as f:
        json.dump(data, f, ensure_ascii=False, indent=2)

    print(f"已更新 {updated_count} 题的金标为人工标注")
```

## 七、修正前后对比

### 修正前（当前状态）

```
检索系统 A 生成金标 → 评估时用检索系统 A 的结果和金标对比 → 指标偏高（自欺欺人）
```

### 修正后（Pooling + 人工标注）

```
多套检索系统生成候选池 → 人工审核确认金标 → 评估时用检索系统结果和人工金标对比 → 指标可信
```

### 指标预期变化

修正后，你大概率会看到：
- **recall 下降**：因为人工金标可能包含系统没找到的相关 chunk
- **precision 可能上升也可能下降**：取决于之前 pipeline 生成金标时的噪声比例
- **整体评估更真实**：这才是你系统真正的水平

---

## 八、总结：三句话记住核心问题

1. **用自己的检索结果当标准答案，等于自己给自己判分，分数当然高** — 这就是信息泄漏。
2. **正确做法是人工标注，或者至少用多套检索合并候选池后再人工审核** — 这就是 Pooling。
3. **不修正的话，你的评估报告没有参考价值** — 因为指标被高估，无法反映真实能力。
