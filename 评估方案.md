> 版本：v3.2 | 更新日期：2026-06-02
> 职责：文档向量化 → 召回文本输出（不含 LLM 生成环节）
> 核心目标：量化召回质量，验证召回文本的质量与下游LLM的兼容性，确保交付下游的文本"可用、够用、不冗余"

每道评估题包含**四级标注**，存储在 `eval_data/eval_dataset.json`：

**字段说明**：

| 字段 | 用途 | 评估层级 |
|------|------|----------|
| `relevant_docs` | 文档名子串匹配，判断文档是否被命中 | Layer 1 |
| `answer_snippets` | 核心术语共现匹配，判断答案关键信息是否覆盖 | Layer 2 |
| `relevant_chunk_ids` | 精确 chunk 匹配，判断 chunk 级召回 | Layer 2（全量模式） |
| `reference_answer` | 标准答案，用于 LLM-as-Judge 或关键词完整性校验 | Layer 4 |

### 2.3 标注流程

```
Step 1: 用当前系统对每道题检索 Top-30
Step 2: 人工审核每个 chunk，标注 relevant / partially_relevant / irrelevant
Step 3: 从 relevant chunks 中提取 answer_snippets（3-5 条核心事实）
Step 4: 综合 answer_snippets 撰写 reference_answer（50-100 字）
Step 5: 交叉审核（至少两人确认）
```

## 三、四层指标体系详解

### Layer 1: 文档级召回（基础门槛）

> 回答：相关文档有没有被检索到？

| 指标 | 公式 | 说明 |
|------|------|------|
| **Doc Recall@K** | 命中的相关文档数 ÷ 总相关文档数 | 核心指标，K=5,10,15 |
| **Doc Precision@K** | 命中的相关文档数 ÷ K | 检索结果纯度 |
| **Doc Hit Rate@K** | 至少命中 1 个相关文档的查询比例 | 最低可用性门槛 |

**匹配规则**：文档名子串包含（`_loose_match`），容忍"认识管理"匹配"认识管理.pdf"等变体。

**合格标准**：

| 指标 | 及格线 | 目标线 |
|------|--------|--------|
| Doc Recall@10 | ≥ 0.85 | ≥ 0.95 |
| Doc Hit Rate@10 | ≥ 0.90 | ≥ 0.98 |

**低于及格线时的对策**：
- 检查该文档是否被正确切分和向量化
- 检查文档名是否在 `source_docs` 元数据中
- 增大检索 Top-K 或降低相似度阈值


### Layer 2: 片段级覆盖（核心指标）

> 回答：召回的文本中是否包含了回答问题所需的关键信息？

| 指标 | 公式 | 说明 |
|------|------|------|
| **Snippet Coverage@K** | 命中的 answer_snippets 数 ÷ 总 snippets 数 | **核心指标**，衡量答案关键信息覆盖 |
| **Chunk Recall@K** | 命中的 relevant_chunk_ids 数 ÷ 总 relevant_chunk_ids 数 | chunk 级精确召回（需 chunk 标注） |
| **Chunk Precision@K** | 命中的 relevant_chunk_ids 数 ÷ K | chunk 级精确率 |
| **Context Utilization** | 召回文本中相关内容占比（估算） | 新增：上下文窗口利用率 |

**Snippet Coverage 计算逻辑**（容忍表述差异，但更强调关键概念覆盖）：

```python
def snippet_coverage(retrieved_text: str, answer_snippets: List[str]) -> float:
    """
    计算召回文本对关键答案片段的覆盖率。
    由于 answer_snippets 来源于人工标注的相关 chunk，
    该指标衡量的是“核心概念是否被检索文本覆盖”，
    而不是完整答案的最终生成正确性。
    """
    if not answer_snippets:
        return 1.0
    hits = 0
    for snippet in answer_snippets:
        core_terms = [
            w for w in jieba.cut(snippet)
            if len(w) >= 2 and w not in STOP_WORDS
        ]
        if not core_terms:
            continue
        matched = sum(1 for t in core_terms if t in retrieved_text)
        n = len(core_terms)
        if matched == 0:
            continue
        if n == 1:
            threshold = 1.0
        elif n == 2:
            threshold = 1.0
        elif n == 3:
            threshold = 2 / 3
        else:
            threshold = 0.6
        if matched >= 2 and matched / n >= threshold:
            hits += 1
    return hits / len(answer_snippets)
```

> 备注：`answer_snippets` 是从人工标注的相关 chunk 中提取的关键事实，
> 所以 `Snippet Coverage` 应与 `Chunk Recall@K` / `Chunk Precision@K` 一起使用，
> 这样才能更准确地判断检索质量。

**Context Utilization 计算逻辑**（新增）：

```python
def context_utilization(retrieved_chunks: List[dict], relevant_chunk_ids: List[str]) -> float:
    """
    上下文窗口利用率：召回文本中真正有用的内容占比。
    用相关 chunk 的字符数 / 总召回字符数。
    """
    total_chars = sum(len(c["content"]) for c in retrieved_chunks)
    if total_chars == 0:
        return 0.0
    relevant_chars = sum(
        len(c["content"]) for c in retrieved_chunks
        if c.get("chunk_id") in relevant_chunk_ids
    )
    return relevant_chars / total_chars
```

**合格标准**：

| 指标 | 及格线 | 目标线 |
|------|--------|--------|
| Snippet Coverage@10 | ≥ 0.70 | ≥ 0.85 |
| Chunk Recall@10 | ≥ 0.60 | ≥ 0.75 |
| Context Utilization | ≥ 0.40 | ≥ 0.60 |

**低于及格线时的对策**：
- Snippet Coverage 低 → 相关 chunk 未被召回或排名太低，增大 Top-K
- Context Utilization 低 → 召回了很多无关内容，提高相似度阈值或增加精排
- Chunk Recall 低 → 切分粒度问题，调整 chunk_size/chunk_overlap


### Layer 3: 排序质量（体验优化）

> 回答：相关结果排得够不够靠前？

| 指标 | 公式 | 说明 |
|------|------|------|
| **MRR@K** | Mean Reciprocal Rank | 第一个相关结果排名的倒数均值 |
| **NDCG@K** | DCG@K ÷ IDCG@K | 排序质量，考虑多级相关性 |
| **MAP@K** | Mean Average Precision | 各位置精确率的均值 |
| **Avg Position** | 所有 relevant chunk 的平均排名 | 整体排序水平 |

**NDCG 相关性分级**：
- `relevant` → rel = 2
- `partially_relevant` → rel = 1
- `irrelevant` → rel = 0

**合格标准**：

| 指标 | 及格线 | 目标线 |
|------|--------|--------|
| MRR@10 | ≥ 0.60 | ≥ 0.75 |
| NDCG@10 | ≥ 0.55 | ≥ 0.70 |

**低于及格线时的对策**：
- MRR 低 → 相关 chunk 存在但排名靠后，调整 RRF 权重，增加相关检索路径的权重
- NDCG 低 → 排序质量整体差，检查是否存在噪声 chunk 排名过高


### Layer 4: 下游兼容性（交付验证）

> 回答：召回文本交给下游同学后，LLM 能否基于此生成正确回答？

这一层是**你与下游同学的接口验证**，确保你输出的召回文本格式、内容、长度都符合下游需求。

#### 4.1 格式兼容性检查（自动化，无需 LLM）

| 检查项 | 要求 | 验证方式 |
|--------|------|----------|
| 字段完整性 | 必须包含 `merged_context`, `source_docs`, `retrieved_chunks`, `qa_hits` | 自动化断言 |
| merged_context 长度 | 不超过 4000 字符 | 自动化统计 |
| 来源标注格式 | `【来源：文档名】` 格式统一 | 正则校验 |
| chunk 内容非空 | 所有 chunk 的 content 非空 | 自动化断言 |
| JSON 可解析 | 输出结果可被 `json.loads` 解析 | 自动化断言 |
| 无乱码 | 不包含 `\ufffd` 等替换字符 | 自动化断言 |

**合格标准**：6/6 全部通过（硬性要求）。

#### 4.2 内容充分性检查（三种模式）

| 模式 | 名称 | 是否需要外部 API | 适用场景 | 结果可信度 |
|------|------|----------------|----------|-----------|
| **A** | 关键词匹配 | 否 | Quick 模式日常迭代 | 中等 |
| **B** | 内置AIjudge | 否 | Quick/Full 均可，零成本 | 较高 |
| **C** | 外部 LLM-as-Judge | 是 | Full 模式最终报告 | 最高 |

**模式 A：关键词匹配（立即可用，无需 API）**

```python
def answer_completeness_keyword(
    merged_context: str,
    reference_answer: str
) -> Dict[str, float]:
    """
    基于关键词的答案完整性评估（无需 LLM）。
    从 reference_answer 中提取核心关键词，检查在 merged_context 中的共现率。
    """
    ref_keywords = set(
        w for w in jieba.cut(reference_answer)
        if len(w) >= 2 and w not in STOP_WORDS
    )
    if not ref_keywords:
        return {"completeness": 0.0, "keyword_hit_rate": 0.0}
    hits = sum(1 for kw in ref_keywords if kw in merged_context)
    return {
        "completeness": hits / len(ref_keywords),
        "keyword_hit_rate": hits / len(ref_keywords),
    }
```

**模式 B：内置AIjudge（内置 AI 评估，零成本）**

```python
def built_in_ai_judge(query: str, merged_context: str, reference_answer: str) -> dict:
    """
    利用内置 AI 评估召回文本的充分性。
    无需调用外部 API，直接在当前环境中运行。
    支持多种 AI 模型（Kimi、GPT、Claude 等），通过统一接口调用。
    """
    prompt = f"""你是一个 RAG 系统检索质量评估专家。请判断以下检索结果是否足以支撑回答用户问题。

用户问题：{query}

检索到的上下文：
{merged_context}

标准参考答案：{reference_answer}

请从以下维度打分(1-5 分)：
1. 相关性(relevance)：检索文本与问题的相关程度
2. 完整性(completeness)：检索文本是否包含回答所需的全部关键信息
3. 准确性(accuracy)：检索文本中是否存在与问题矛盾或误导的内容
4. 简洁性(conciseness)：检索文本中无关内容的比例(5=无冗余)

输出 JSON：{{"relevance":X, "completeness":X, "accuracy":X, "conciseness":X, "can_answer":true/false, "missing_info":"..."}}"""

    # 在当前 AI 环境中执行评估
    response = ai_call(prompt)
    return json.loads(response)
```

## 四、检索鲁棒性测试

### 4.1 为什么需要鲁棒性测试

真实用户的提问不会像测试题一样规范。用户可能用同义词、缩写、错别字甚至中英混合来提问。如果系统只在"标准问题"上表现好，在真实场景下就会出问题。

### 4.2 测试矩阵

从 55 题中抽取 15 道典型题，每题生成 4 种变体，共 60 题：

| 变体类型 | 示例 | 测试目的 |
|----------|------|----------|
| 原文 | "什么是领导力" | 基线 |
| 同义改写 | "领导力的定义是什么" / "什么叫领导力" | 语义理解能力 |
| 术语缩写 | "SWOT分析" → "SWOT" | 缩写解析能力 |
| 中英混合 | "OKR工作法" → "OKR" | 中英混合查询 |
| 口语化表达 | "霍桑实验" → "那个著名的工厂实验" | 模糊查询能力 |

### 4.3 鲁棒性指标

| 指标 | 计算方式 | 合格标准 |
|------|----------|----------|
| **Robustness Score** | 变体题的平均 Recall ÷ 原文题的 Recall | ≥ 0.85 |
| **Degradation Rate** | 变体中 Recall 下降超过 20% 的题目比例 | ≤ 0.20 |

**解读**：
- Robustness Score = 0.95 → 系统对变体几乎不受影响，鲁棒性优秀
- Robustness Score = 0.70 → 系统对变体敏感，需要优化查询扩展策略
- Degradation Rate = 0 → 没有题目出现显著退化

### 4.4 鲁棒性失败对策

| 退化类型 | 原因 | 对策 |
|----------|------|------|
| 同义改写退化 | 向量检索对同义词不敏感 | 增强查询扩展的同义词库 |
| 缩写退化 | 知识库只有全称没有缩写 | 在查询扩展中补充缩写↔全称映射 |
| 口语化退化 | 口语词与学术词向量距离大 | 减少 BM25 权重，增加 Dense 权重 |
| 中英混合退化 | 跨语言检索能力不足 | 在查询扩展中统一转换为中文 |


## 五、失败模式分类体系

### 5.1 六类失败模式

不再只是"列出失败案例"，而是将每个失败案例归入以下六类之一，每类都有明确的修复方向：

| 编号 | 失败模式 | 症状 | 诊断方法 | 修复方向 |
|------|----------|------|----------|----------|
| **F1** | 查询词失配 | 用户用词与文档用词不一致，BM25 和 Dense 都未命中 | 检查 query 关键词是否出现在相关文档中 | 扩充查询扩展同义词库，加入领域术语映射 |
| **F2** | 文档覆盖不足 | 相关文档的 chunk 未被索引或向量化 | 检查相关文档是否在向量库中 | 重新导入文档，检查清洗流程 |
| **F3** | 切分丢失 | 答案被切分边界切断，一个 chunk 包含不完整信息 | 检查 relevant chunk 的相邻 chunk | 增大 chunk_size 或 overlap |
| **F4** | 排序下沉 | 相关 chunk 存在但排名太低（>10） | 检查相关 chunk 在 Top-30 中的位置 | 调整 RRF 权重，增加该 chunk 所在检索路径的权重 |
| **F5** | 信息分散 | 答案需要多个 chunk 的信息拼接，但仅命中了部分 | 检查召回 chunk 的信息完整性 | 增大 Top-K，或启用父 chunk 扩展 |
| **F6** | 噪声优势 | 不相关的 chunk 排名靠前，挤掉了相关 chunk | 检查 Top-10 中 irrelevant chunk 的类型 | 提高相似度阈值，或增加 Cross-Encoder 精排 |

| qa_heavy | 1.0 | 0.8 | 0.8 | 2.0 | 偏 QA → 对事实类友好 |

**关键指标关注**：MRR, NDCG, 按 query_type 分解的 Recall

#### E6: 向量清洗

| 配置 | 去重 | 质量筛选 | 假设 |
|------|------|----------|------|
| no_clean | 否 | 否 | 无清洗基线 |
| dedup_only | 是 | 否 | 去重减少冗余 |
| dedup_quality | 是 | 是 | **当前配置（基线）** |

**关键指标关注**：Context Utilization, Chunk Precision, 总向量数

#### E7: Cross-Encoder 精排

| 配置 | USE_CROSS_ENCODER | CROSS_ENCODER_TOP_K | 假设 |
|------|-------------------|---------------------|------|
| no_rerank | False | - | 无精排基线 |
| rerank_top10 | True | 10 | 精排Top-10，减少计算量 |
| rerank_top15 | True | 15 | **当前配置（基线）** |

**说明**：Cross-Encoder（`BAAI/bge-reranker-large`）在 RRF 融合 + MMR 重排之后，对候选结果进行二次精排。它将查询和文档拼接输入，计算精确相关性分数，比单纯向量相似度更准确。代价是每次查询增加约 0.5-1 秒延迟。

**关键指标关注**：MRR, NDCG, Snippet Coverage（排序改善后前几条更相关）

#### E8: BM25 章节标题 Boost

| 配置 | boost系数 | 假设 |
|------|-----------|------|
| no_boost | 0 | 无章节boost基线 |
| mild_boost | 0.10 | 轻微加成 |
| current_boost | 0.15 | **当前配置（基线）** |

**说明**：在 BM25 检索中，对章节标题与查询词重叠的 chunk 给予分数加成（1 + overlap_count * boost系数）。章节标题通常包含文档核心概念，匹配标题的 chunk 更可能包含用户所需信息。

**关键指标关注**：Doc Recall, MRR, 按 query_type 分解的 Recall

#### E9: MMR 动态参数

| 配置 | λ策略 | 假设 |
|------|-------|------|
| fixed_lambda | λ=0.45 固定 | 固定参数基线 |
| dynamic_lambda | 按问题类型动态调整 | **当前配置（基线）** |
| aggressive_diverse | λ=0.30 全局 | 更高多样性 |

**说明**：动态λ根据问题类型调整 MMR 的相关性/多样性平衡：
- 对比类问题：λ=0.30（更高多样性，需要多角度信息）
- 定义类问题：λ=0.60（更高相关性，需要精准定义）
- 默认：λ=0.45（平衡模式）

**关键指标关注**：Snippet Coverage, 按 query_type 分解的 Recall, 跨文档召回率

## 八、缓存与索引管理

### 8.1 为什么需要缓存管理

评估过程中，**向量索引的构建是最耗时的环节**（约 2-3 分钟）。如果每次运行评估都重新构建索引，不仅浪费时间，还会阻碍快速迭代。

**核心原则**：
- **切分策略或清洗策略改变** → 必须重建索引（chunk 变了，向量也变了）
- **检索策略、Top-K、RRF 权重、查询扩展改变** → 无需重建索引（只影响检索阶段）

### 8.2 缓存文件清单

| 缓存文件 | 路径 | 构建耗时 | 是否必须保留 | 重建触发条件 |
|----------|------|----------|-------------|-------------|
| FAISS 小粒度索引 | `vector_store/small/index.faiss` | ~2-3min | 必须保留 | 切分策略/清洗策略改变 |
| FAISS 大粒度索引 | `vector_store/large/index.faiss` | ~2-3min | 必须保留 | 切分策略/清洗策略改变 |
| FAISS 小粒度元数据 | `vector_store/small/metadata.json` | ~10s | 必须保留 | 切分策略/清洗策略改变 |
| FAISS 大粒度元数据 | `vector_store/large/metadata.json` | ~10s | 必须保留 | 切分策略/清洗策略改变 |
| FAISS 小粒度向量 | `vector_store/small/embeddings.npy` | ~10s | 必须保留 | 切分策略/清洗策略改变 |
| FAISS 大粒度向量 | `vector_store/large/embeddings.npy` | ~10s | 必须保留 | 切分策略/清洗策略改变 |
| BM25 索引 | 内存中构建 | ~30s | 运行时构建 | 每次启动重建（很快） |
| QA 对数据 | `qa_store/qa_pairs.json` | ~5s | 必须保留 | QA 对更新时 |
| QA 向量 | `qa_store/qa_embeddings.npy` | ~30s | 必须保留 | QA 对更新时 |
| 实验预构建索引 | `vector_store/experiments/` | ~5min/组 | 建议保留 | 切分策略实验时 |

### 8.3 缓存加载策略

```python
def load_system_for_eval(force_rebuild=False):
    """
    加载系统用于评估，优先复用缓存。
    只有在切分策略或清洗策略改变时才需要重建。
    """
    pipeline = RAGPipeline()

    if not force_rebuild and pipeline._check_cache():
        print("[评估] 复用缓存加载向量库...")
        pipeline._load_from_cache()
        # BM25 索引在内存中，每次运行都需要重建，但很快 (~30s)
        pipeline.retriever.bm25 = BM25Retriever(pipeline.all_chunks)
        return pipeline
    else:
        print("[评估] 缓存不存在或强制重建，重新构建知识库...")
        pipeline.build_knowledge_base(force_rebuild=True)
        return pipeline
```

### 8.4 实验索引预构建

对于 E1（切分策略）和 E6（向量清洗）实验，建议**预构建索引**避免重复构建：

```python
# 预构建各切分策略的索引
for size, overlap in [(400, 40), (600, 60), (800, 80), (1200, 120)]:
    config_name = f"chunk_{size}"
    pipeline = RAGPipeline(chunk_size=size, chunk_overlap=overlap)
    pipeline.build_knowledge_base()
    pipeline.save_to_cache(f"vector_store/experiments/{config_name}/")

# 后续实验直接加载
pipeline = RAGPipeline()
pipeline.load_from_cache("vector_store/experiments/chunk_400/")
```

### 8.5 缓存清理建议

| 场景 | 操作建议 |
|------|----------|
| 日常迭代（改检索参数） | 保留所有缓存，直接加载 |
| 切分策略实验 | 预构建 4 组索引到 `experiments/`，保留复用 |
| 清洗策略实验 | 预构建 3 组索引到 `experiments/`，保留复用 |
| 磁盘空间不足 | 可删除 `experiments/`，保留主索引 |
| 代码升级后 | 检查缓存版本兼容性，必要时重建 |

## 十一、实施路线图

```
Phase 1: 数据准备（优先）
  ├── 编写 eval_data/eval_dataset.json（55 题 + 标注）
  ├── 标注 answer_snippets（每题 3-5 条）
  ├── 编写 reference_answer（每题 50-100 字）
  └── 标注 relevant_chunk_ids（可选，Full 模式用）

Phase 2: 核心评估框架
  ├── 实现 src/eval/dataset.py（加载与验证）
  ├── 实现 src/eval/metrics.py（四层指标）
  ├── 实现 src/eval/reporter.py（报告生成）
│   ├── 实现 src/eval/built_in_ai_judge.py（内置 Judge）
  └── 实现 src/eval_quick.py（Quick 模式入口）

Phase 3: 运行首次基线评估
  ├── 跑 Quick 模式 → 获取基线指标
  ├── 跑鲁棒性测试 → 获取鲁棒性基线
  ├── 分析失败模式 → 识别优化方向
  └── 与下游同学对齐输出格式

Phase 4: 优化迭代
  ├── 根据失败模式分类调整参数
  ├── 每次调整后跑 Quick 模式验证
  └── 与下游同学联调，确认接口契约

Phase 5: Full 模式评估
  ├── 跑 6 组控制变量实验
  ├── 跑 内置AIjudge（零成本，必跑）
  ├── 跑外部 LLM-as-Judge（如有 API）
  ├── 生成全部可视化图表
  └── 输出最终评估报告 → 写入课程报告
```


## 十二、合格标准汇总

| 层级 | 指标 | 及格线 | 目标线 | 说明 |
|------|------|--------|--------|------|
| L1 | Doc Recall@10 | ≥ 0.85 | ≥ 0.95 | 基础门槛 |
| L1 | Doc Hit Rate@10 | ≥ 0.90 | ≥ 0.98 | 最低可用性 |
| L2 | Snippet Coverage@10 | ≥ 0.70 | ≥ 0.85 | **核心指标** |
| L2 | Chunk Recall@10 | ≥ 0.60 | ≥ 0.75 | 需 chunk 标注 |
| L2 | Context Utilization | ≥ 0.40 | ≥ 0.60 | 上下文窗口利用率 |
| L3 | MRR@10 | ≥ 0.60 | ≥ 0.75 | 排序质量 |
| L3 | NDCG@10 | ≥ 0.55 | ≥ 0.70 | 排序质量 |
| L4 | Can-Answer Rate (内置AIjudge) | ≥ 0.80 | ≥ 0.90 | 端到端可用性 |
| L4 | 格式兼容性 | 100% | 100% | 硬性要求 |
| R | Robustness Score | ≥ 0.85 | ≥ 0.92 | 鲁棒性 |
| R | Degradation Rate | ≤ 0.20 | ≤ 0.10 | 退化率 |
| — | 失败案例数 | ≤ 10 | ≤ 5 | 55 题中 Recall<0.5 的题数 |
